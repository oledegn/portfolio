HTML & CSS


Det er gået op for mig at ”webdesignere er nødt til at kende til HTML og CSS i et vist omfang, da det er selve det grundlæggende sprog i det medie designeren arbejder med, ikke mindst pga et stigende behov for at tilpasse designet til forskellige medier og skærmstørrelser” (J.N. Robbins: Learning Web Design, p 10)


HTML (Hyper Text Markup Language):

Består af en række tags, som tilsammen udgør selve ”sprog”-strukturen for en hjemmeside. Disse tags kan være semantiske eller ikke-semantiske, hvor sidstnævnte, f.eks. "div" (inddeling) og "span" (sætningsindhold), ikke siger noget om indholdet. Det gør de semantiske tags til gengæld, som er bedre at bruge, og kunne være f.eks.: "header" (sidehoved), "footer" (sidefod), "section" (sektion), "article" (tekst) eller h1 (stor overskrift). Der indledes med et start-tag "tagname" og afsluttes med et end-tags "/tagname".

CSS (Cascading Style Sheets):

Hvor HTML bruges til at beskrive indholdet på en hjemmeside, bruges CSS til at beskrive hvordan dette skal se ud – man ”styler” altså indholdet (farver, fonte, baggrundsbilleder, side-layout osv.). I CSS bruger man ”tuborg-tags”{ } til at omkranse den styling man ønsker. CSS-sproget kan være integreret på HTML-siden eller (smartere) laves på sin egen side, som så linkes til HTML-dokumentet.

Eksempel på CSS: body { font: italic; color: black } = Alt indholdet imellem body-tags i HTML får nu fonten ”Italic” og farven ”sort”.


CSS Grid:

Er det nyeste indenfor CSS-standarden, hvor man placerer sine elementer i et gridsystem (2 akser). Det bygger delvist videre på flexbox (kun 1 akse), og afløser til dels også dette. Også responsive webdesign er en nyere landvinding indenfor styling, hvor man kan tilpasse sit CSS-design til forskellige skærmstørrelser.

FAGUDTRYK OG KODEPROJEKTER


Kode-editor:

Koden for HTML og CSS skrives ind i en editor, og der findes en del af slagsen. Jeg har afprøvet Atom og Microsoft Visual Studio (VSC), og bruger pt. sidstnævnte.


Client/server:

Selve fundamentet for brugen af internettet:

Clienten efterspørger data, og kan være en browser (f.eks. Safari, Microsoft Edge eller Google Chrome). Serveren er ”computeren”, der hoster ”sitet” eller de data der ønskes (f.eks. IIS eller Apache) - vi bruger Danias servere, der er placeret i Nordjylland.

Man benytter desuden FTP (File Transfer Protocol) til at flytte data frem og tilbage.


User interface værktøjer:

Et nyere fænomen er nogle design-værktøjer, hvor man kan pre-designe layouts før selve programmeringen af websitet. Man kan afprøve sine designs i forskellige skærmstørrelser på test-niveau, og efterfølgende endda eksportere kode og data til den endelige kodning. Vi har afprøvet Adobe XD (se Design-redesign).


IP-adresse (Internet Protocol):

Enhver computer og andre enheder på nettet får tildelt et nummer, f.eks. 123.45.678.90. IP er protokoller, som har større relevans for computer-software end mennesker. De er knudepunkter, der forbinder verden.


URL (uniform ressource locator):

Er websideadresser, f.eks. http://www.eksempel.dk/2021/endeligversion.html - hvor:

http:// (HyperText Transfer Protocol) = indledende protocol, hvor serveren går i ”web mode”.

https:// = secure protocol, hvor brugerens data er mere sikre (bruges af efterhånden 60% websites)

www = efterhånden en konvention, men den kan ofte undlades

eksempel.dk = domænenavnet

/2021/ = root-directory (efter det første slash /)

endeligversion.html = selve dokumentet

VERSIONSSTYRING


Som versionsstyringssystem ifm. denne opgave har jeg benyttet GitHub, der er en standard indenfor backup af kode i dag. Hermed undgår man at komme til at slette noget ved en fejl, og desuden kan man gå tilbage til tidligere backup-versioner, hvis man har fortrudt.

Man tilgår sine filer på github.com eller måske lettere via GitHub desktop, der kan downloades til computeren.

Jeg har brugt GitHub i forbindelse med denne opgave, idet jeg har oprettet en destinationsmappe kaldet ”portfolie”, og dagligt koblet på GitHub desktop og gemt det sidst nye.



Her er en forklaring på de forskellige begreber i GitHub:

  • Repository: Er selve hovedmappen som oprettes først, når man navngiver sit projekt
  • Commit: Når man gemmer sit projekt, trykkes commit, og versionen lige nu gemmes
  • Branch ( = gren). Main branch er ”hovedgrenen”, som man arbejder på, men man kan oprette endnu en branch (= sidegren, der tager udgangspunkt i main branch), hvis jeg ønsker at eksperimentere i en ny retning, uden at ødelægge det oprindelige
  • Pull Request: Når man åbner pull request, foreslår man ændringer, f.eks. til gennemsyn hos andre, som kan inviteres til at se dem, eller de kan flette dette ind i deres eget projekt. Da denne delingsmulighed er meget benyttet, tangerer GitHub et socialt medie
  • Merge: Hvis man ønsker at flette ændringerne i Pull Request sammen med f.eks. sit Main branch
